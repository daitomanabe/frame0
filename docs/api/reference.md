# FRAME0 API Reference

FRAME0 is API-first through command-line contracts, schema exports, and native
C ABI boundaries. This document describes the v0.1 public surface as implemented
in the repository. Read [Compatibility](compatibility.md) before changing public
schemas, CLI JSON, NDJSON events, or native ABI headers.

## CLI Contract

Every operational CLI command accepts `--json` where applicable. Automation
should prefer JSON/NDJSON output over human text.

```bash
frame0 doctor --json
frame0 inspect examples/hello_shader/scene.yaml --json
frame0 graph examples/hello_shader/scene.yaml --json
frame0 graph diff before.yaml after.yaml --json
frame0 run examples/hello_shader/scene.yaml --dry-run --json
frame0 run examples/hello_shader/scene.yaml --events ndjson --frames 3
frame0 render examples/headless_render/scene.yaml --frames 60 --json
frame0 snapshot runtime --scene examples/audio_reactive/scene.yaml --json
frame0 docs index --json
frame0 docs examples --json
frame0 schema list --json
frame0 schema export all --json
frame0 examples list --json
frame0 examples launch projection_mapping --frames 120 --out runs/examples/projection_mapping --json
frame0 examples launch-all --frames 24 --out runs/examples --json
```

Command errors are process failures with a readable stderr message. Structured
diagnostics generated by FRAME0 use `schemas/error.schema.json`.

## Scene Manifest API

Scene manifests are YAML or JSON documents loaded by `frame0 inspect`, `graph`,
`run`, `render`, and `snapshot`. The current schema is exported by:

```bash
frame0 schema export scene --json
```

Core fields:

| Field | Purpose |
| --- | --- |
| `name` | Stable scene name for diagnostics and generated reports |
| `version` | Scene contract version |
| `runtime` | Runtime target, currently `frame0` |
| `permissions` | Camera, microphone, network, filesystem, and native permission intent |
| `clock` | Manual, monotonic, audio, or fixed-step timing policy |
| `nodes` | Named processing/render/input/output resources |
| `outputs` | Screen, file, extension, audio, or network output bindings |
| `metadata` | Optional project and tool metadata |

FRAME0 validates references between nodes and outputs and emits diagnostics with
machine-readable codes.

## Graph API

`frame0 graph <scene> --json` returns a graph object with nodes, edges,
diagnostics, and topological order. Use it to inspect dependencies before
running a scene or to compare revisions:

```bash
frame0 graph diff before.yaml after.yaml --json
```

Graph output is intended for tooling, visual editors, build systems, and AI
agents. Do not parse human output.

## Runtime Event API

The runtime event stream is line-delimited JSON:

```bash
frame0 run examples/hello_shader/scene.yaml --events ndjson --frames 3
```

Each line is an event packet with a stable event name, frame index or timestamp
when available, and event-specific data. This is the preferred integration point
for AI monitoring, regression capture, benchmark tooling, and headless replay.

## Runtime Snapshot API

`frame0 snapshot runtime --json` returns the current resource registry,
resolved devices, graph summary, timing policy, and permissions. It is the
lowest-friction way to inspect what a scene will require before execution.

## Resource API

Resources are inspectable runtime objects derived from the scene graph and
device model.

```bash
frame0 resources list --scene examples/hello_shader/scene.yaml --json
frame0 resource get node.color_shader --scene examples/hello_shader/scene.yaml --json
```

`resource set` currently validates mutation requests and returns a placeholder
result. Real mutation transport is reserved for the runtime supervisor and IPC
layer.

## Device API

The v0.1 scaffold exposes mock devices for video input, audio input, OSC,
screen capture, and extension-style outputs:

```bash
frame0 devices list --json
frame0 devices modes device.video_input.mock.0 --json
```

Native/hardware SDKs should appear as adapter-backed devices rather than
leaking vendor SDK objects into scene manifests.

## Plugin Manifest API

Plugin manifests describe adapter identity, dynamic library entrypoints,
capabilities, isolation, and permissions.

```bash
frame0 plugins verify plugins/mock/plugin.yaml --json
frame0 plugins inspect io.frame0.mock.sdk --json
```

Third-party packages should keep runtime capabilities explicit and minimal.
FRAME0 expects native plugins to run out of process.

## Native Plugin ABI

The public native plugin boundary is the C header:

```text
native/frame0_plugin_c_api/frame0_plugin_api.h
```

Adapters must not expose C++ ABI, exceptions, STL types, Objective-C runtime
objects, or vendor SDK handles across this boundary. Use JSON control messages
for adapter-specific metadata where the ABI does not define a typed packet yet.

## Native External ABI

Native externals use:

```text
native/frame0_external_c_api/frame0_external_api.h
```

The external template demonstrates a C implementation and a C++ wrapper behind
that stable C ABI:

```text
templates/external-c/
templates/external-cpp/
```

## ML Adapter API

ML is represented as adapter-backed inference, not as core runtime logic.
Relevant schemas:

- `schemas/ml_model.schema.json`
- `schemas/inference_packet.schema.json`

Mock commands:

```bash
cargo run -p frame0_plugin_host -- ml-describe plugins/mock_ml/plugin.yaml --json
cargo run -p frame0_plugin_host -- ml-infer plugins/mock_ml/plugin.yaml --model mock_classifier --json
```

Future Core ML, MPSGraph, ANE, or vendor runtimes should replace adapter
internals while preserving the runtime-facing model and inference packets.

## Addon API

Addon packages are self-contained directories with an addon manifest, examples,
tests, and native/plugin assets when needed. See:

- `docs/addons/authoring-guide.md`
- `docs/addons/registry.md`
- `templates/addon-rust/`

Addon verification starts with:

```bash
scripts/verify_addon_package.sh templates/addon-rust
```

Repository example verification starts with:

```bash
scripts/verify_examples.sh
```
